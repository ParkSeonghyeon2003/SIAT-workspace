Python 모듈(Module)·패키지(Package)

1) 모듈(Module) 정의

정의

모듈(Module)은 파이썬 코드가 들어 있는 하나의 파일(.py)입니다.
함수, 클래스, 변수 등을 한 파일에 모아두고, 다른 파일에서 import로 불러 사용합니다.

왜 쓰나요?
  • 재사용이 쉬워집니다.
  • 코드 분리로 관리가 쉬워집니다.
  • 협업에서 파일 단위로 역할을 나누기 좋습니다.

모듈의 기본 예

# calc.py (모듈 파일)
def add(a, b):
    return a + b

PI = 3.141592

class Person:
    def __init__(self, name):
        self.name = name

# main.py (사용 파일)
import calc

print(calc.add(2, 3))
print(calc.PI)

p = calc.Person("Kim")
print(p.name)

__name__ 과 모듈 실행

파이썬은 파일을 실행하거나 import 할 때 __name__ 변수에 값을 넣습니다.
직접 실행하면 __name__ == "__main__" 입니다.
import되면 __name__은 모듈 이름이 됩니다. 

# tool.py
def run():
    print("run() 실행됩니다.")

if __name__ == "__main__":
    print("이 파일을 직접 실행했습니다.")
    run()

# main.py
import tool
tool.run()

2) 모듈의 종류

종류 설명 예시
표준 라이브러리(Standard Library) 파이썬 설치 시 기본 포함된 모듈이며, import 해서 사용합니다. math, random, datetime, pathlib, json, sys, os, time
서드파티(Third-party) pip로 설치해서 쓰는 외부 모듈(외부 라이브러리)입니다. requests, pandas, numpy, flask
사용자 정의(User-defined) 직접 만든 .py 파일 모듈입니다. calc.py, utils.py
참고: 내장 객체(Builtins)
- 내장 객체는 print(), len(), list, int, Exception 처럼 import 없이 바로 사용하는 파이썬의 기본 구성 요소입니다.
- 엄밀히 말하면 '모듈'이라기보다 파이썬이 실행될 때 자동으로 로드되는 내장(builtins) 네임스페이스(변수명, 함수명이 저장되어 있는 공간)에 속한 객체들입니다.
- 반면, 표준 라이브러리(sys, math 등)는 파이썬이 제공하지만 반드시 import를 해야만 사용할 수 있다는 차이점이 있습니다.
정리: 외부 라이브러리(서드파티)는 “기본 설치에 없음 → 필요할 때 pip로 설치”하는 패키지입니다. pip의 개념/설치 흐름은 4번 탭(pip)에서 다룹니다.

3) import 사용법(가져오기)

1) 모듈 전체 import

import math
print(math.sqrt(16))

2) 별칭(alias) 사용

import math as m
print(m.sqrt(16))

3) 특정 요소만 import

from math import sqrt, pi
print(sqrt(16))
print(pi)

4) 전체 import(*)는 주의해서 사용

from module import *는 이름 충돌이 발생할 수 있어 학습용이 아니라면 권장하지 않습니다. 
from math import *
print(sqrt(16))

3) 모듈 검색 경로 (sys.path)

파이썬은 import 문을 실행하면 아래와 같은 우선순위에 따라 모듈을 검색합니다.

  1. sys.modules (캐시 확인): 이미 로드된 모듈인지 먼저 확인합니다. (메모리에 있으면 즉시 반환)
  2. Built-in Modules (내장 모듈): sys, time과 같이 파이썬 인터프리터에 컴파일되어 포함된 핵심 모듈을 찾습니다.
  3. sys.path (경로 리스트): 리스트에 등록된 경로들을 순서대로 탐색합니다.
    • 현재 실행 위치: 스크립트 파일이 있는 폴더 (가장 높은 우선순위)
    • 표준 라이브러리 경로: 파이썬 설치 시 기본 제공되는 모듈 폴더
    • site-packages: pip로 설치한 외부(Third-party) 라이브러리 경로
파일명 중복 주의 (Shadowing)
현재 실행 위치가 표준 라이브러리보다 우선순위가 높습니다. 만약 내 폴더에 math.py나 random.py라는 파일을 만들면, 파이썬 기본 모듈 대신 내가 만든 파일이 로드되어 오류가 발생할 수 있습니다. 
import sys
for p in sys.path:
    print(p)

4) pip (외부 라이브러리 설치/관리)

정의

pip(Pip Installs Packages)는 파이썬 패키지(라이브러리)를 설치/업데이트/삭제하는 패키지 관리 도구입니다.
외부 라이브러리(서드파티)는 기본 설치에 없기 때문에, 보통 pip로 설치합니다.

1) pip 기본 확인

pip --version             # 시스템 환경 변수(PATH)에 등록된 pip 파일을 찾아 실행합니다.
python -m pip --version   # 특정 파이썬 인터프리터를 먼저 실행한 뒤, 그 파이썬에 포함된 pip 모듈을 실행합니다.

2) 자주 쓰는 pip 명령어

# 설치
pip install requests

# 특정 버전 설치
pip install requests==2.32.3

# 업그레이드
pip install --upgrade requests

# 삭제
pip uninstall requests

# 설치 목록
pip list

# 패키지 정보
pip show requests

3) requirements.txt로 한 번에 설치하기

# requirements.txt 예
requests==2.32.3
pandas==2.2.2
numpy==2.1.0

pip install -r requirements.txt

4) pip install requests 동작 과정(설치 흐름)

목표

pip install requests를 실행하면 pip가 내부적으로 어떤 순서로 동작하는지 이해합니다.
검색 → 의존성 해결 → 다운로드 → site-packages 설치 흐름입니다. 

pip install requests
# 또는(권장)
python -m pip install requests
단계 pip 내부에서 일어나는 일 설명
1 pip 실행(환경 결정) 현재 파이썬(전역/가상환경) 기준으로 설치 위치가 결정됩니다.
2 PyPI(Python Package Index) 접속 pip는 기본 저장소인 PyPI에서 패키지를 찾습니다(보통 인터넷 필요)
Python Package Index는 전 세계 개발자들이 만든 패키지들이 저장되어 있는 '온라인 저장소(웹사이트)'입니다.
주소는 pypi.org입니다.
3 메타데이터 조회 버전, 호환성, 배포 파일(.whl/.tar.gz), 의존성 정보를 가져옵니다.
4 의존성 분석 requests가 필요로 하는 패키지들을 함께 설치 대상으로 잡습니다.
5 파일 다운로드 보통 wheel(.whl)을 우선 선택해 다운로드합니다.
6 site-packages 설치 현재 환경의 site-packages에 설치되고 .dist-info도 생성됩니다.
7 의존성 반복 설치 필요한 의존성 패키지들도 같은 흐름으로 함께 설치합니다.
8 완료 설치가 성공하면 Successfully installed 메시지가 출력됩니다.

5) 설치 후 import가 되는 이유

설치가 끝나면 패키지가 site-packages에 들어가고, 파이썬은 sys.path를 검색해서 모듈을 찾기 때문에 import가 가능합니다. 
python -c "import requests; print(requests.__version__)"

5) 패키지(Package) 개념

정의

패키지(Package)는 여러 모듈(.py 파일)을 폴더(디렉터리)로 묶어 관리하는 구조입니다.
쉽게 말해 모듈들의 묶음 폴더이며, “폴더 안에 모듈이 있고 import로 가져다 쓰는 단위”입니다.

패키지 구조 예

project/
  main.py
  mypkg/
    __init__.py
    calc.py
    io/
      __init__.py
      reader.py
포인트
- mypkg 폴더 자체가 import 대상(패키지)이 됩니다.
- 그 안의 calc.py, io/reader.py는 모듈입니다.
- 패키지/모듈을 잘 나누면 “기능별로 파일을 정리”하기 쉬워집니다.

__init__.py 를 왜 쓰나요?

핵심 역할
  • 패키지 초기화 코드를 실행하는 곳입니다. (패키지가 처음 import될 때 1번 실행됩니다.)
  • 패키지의 공개 API(외부에 보여줄 함수/클래스)를 정리할 수 있습니다.
  • from 패키지 import ... 형태에서 어떤 이름을 내보낼지 제어할 수 있습니다.
  • (선택) 서브모듈을 미리 import 하여 사용 편의성을 높일 수 있습니다.
참고(파이썬 버전 차이)
- 예전에는 “패키지로 인식”되려면 __init__.py가 필수였습니다.
- 요즘 파이썬은 네임스페이스 패키지 개념으로 __init__.py가 없어도 패키지가 될 수 있지만,
명확성과 관리 때문에 보통 __init__.py를 둡니다.

1) “패키지가 import될 때” __init__.py 는 자동 실행됩니다

# mypkg/__init__.py
print("mypkg 패키지가 import 되었습니다.")

# main.py
import mypkg
# 실행하면: "mypkg 패키지가 import 되었습니다." 출력
즉, import mypkg를 하면 파이썬은 내부적으로
“mypkg 폴더를 패키지로 로드 → mypkg/__init__.py를 실행” 흐름으로 처리합니다.

2) 공개 API(외부에 제공할 함수/클래스) 묶기

패키지를 사용하는 사람이 mypkg.calc.add처럼 깊게 들어가지 않고,
from mypkg import add처럼 바로 쓰게 만들 수 있습니다. 

# mypkg/calc.py
def add(a, b):
    return a + b

def sub(a, b):
    return a - b

# mypkg/__init__.py
from .calc import add   # 외부에 add만 "대표 기능"으로 공개

# (선택) * import 제어용
__all__ = ["add"]

# main.py
from mypkg import add
print(add(10, 20))  # 30

3) __all__ 은 언제 의미가 있나요?

__all__은 주로 from mypkg import *에서 “어떤 이름을 내보낼지”를 정합니다.
즉, * import를 제어하는 목록입니다. 

# mypkg/__init__.py
from .calc import add, sub
__all__ = ["add"]   # * import 시 add만 공개

# main.py
from mypkg import *
print(add(1, 2))   # 가능
print(sub(1, 2))   # NameError 가능 (sub를 내보내지 않았기 때문)
수업/실무에서는 from 패키지 import *를 권장하지 않지만,
라이브러리 형태로 배포할 때 “공개 범위 정리”에 __all__이 도움이 됩니다.

4) (선택) 서브모듈을 미리 연결해 사용 편의성 높이기

사용자가 mypkg.io.reader를 매번 직접 import하지 않고도 쓸 수 있게
__init__.py에서 연결해 줄 수 있습니다. 

# mypkg/io/reader.py
def read_text():
    return "data"

# mypkg/__init__.py
from .io.reader import read_text
__all__ = ["read_text"]

# main.py
from mypkg import read_text
print(read_text())

상대 import(패키지 내부)

# mypkg/io/reader.py
from ..calc import add  # ..은 상위 폴더(mypkg)를 의미합니다.

def read_and_add(a, b):
    return add(a, b)

# main.py  
from mypkg.io.reader import read_and_add
print(read_and_add(10, 20))
주의
패키지 내부 파일(예: mypkg/io/reader.py)을 “직접 실행”하면 상대 import가 실패할 수 있습니다.
패키지는 보통 루트 엔트리(main.py)에서 실행하고, 내부는 import로 연결합니다.

자주 하는 실수(짧게)

  • __init__.py 에 너무 무거운 작업을 넣으면 import가 느려집니다. (파일 읽기/네트워크 호출 등은 지양)
  • 패키지 이름과 같은 파일명 충돌: 예) mypkg.py 파일을 만들면 import mypkg가 꼬일 수 있습니다.
  • __all__은 “* import 제어용”이라는 점을 혼동하기 쉽습니다.
